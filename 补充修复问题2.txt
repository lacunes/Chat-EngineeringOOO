一、修复 Provider 管理页按钮无效问题

当前 Web 模型管理页里的“编辑 / 测试 / 清除 / 禁用 / 删除”等按钮疑似无效。请重点检查 `web/routes/providers.py` 和 `web/templates/providers.html`。

要求：

1. 不要把 provider name 放在 URL path 里，因为 provider name 或 model 可能包含 `/`、`[]`、`()`、`@`、`:`、空格等特殊字符。
2. 所有 Provider 操作统一使用 POST body 传递 provider 名称，例如：

```text
POST /providers/toggle
POST /providers/clear
POST /providers/test
POST /providers/edit
POST /providers/delete
```

3. 后端从 `request.form.get("name")` 或 JSON body 读取 provider 名称。
4. 前端按钮使用普通 form 或稳定的 fetch 请求，不要依赖错误的路由参数。
5. 每个 form 必须带 CSRF token。如果使用 fetch，必须在 header 中带 `X-CSRF-Token`。
6. 每个按钮执行后必须有明确反馈：

   * 成功：flash 成功消息；
   * 失败：flash 失败原因；
   * 测试失败时显示 provider、model、base_url、错误类型和错误摘要。
7. 删除 Provider 必须二次确认。
8. 清除按钮应只清除该 provider 的 failures、cooldown、exhausted、last_failure_reason 等状态，不得修改 memory、world 或 session。
9. 编辑保存时必须使用原子写入保存 `providers.yaml`。
10. 启用/禁用 provider 后，必须触发 router reload，Web 页面刷新后状态应立即正确。

请完成后自测：

```text
点击测试 → 能看到成功或失败原因
点击编辑 → 能看到当前 provider 的已有配置
保存编辑 → providers.yaml 正确更新
点击禁用/启用 → provider enabled 状态正确变化
点击清除 → cooldown/failures/exhausted 被清空
点击删除 → provider 从 providers.yaml 删除，provider_state 同步清理或归档
provider 名称包含 []、/、空格时仍然正常
model 名称包含 [] 时仍然正常
```

二、修复模型显示不一致：顶栏、仪表盘、/status 必须统一读取 LLMRouter

当前 Web 顶栏、仪表盘卡片、Telegram `/status` 显示的模型可能不一致。例如顶栏显示 MiniMax-M3，但仪表盘显示 deepseek-v4-flash。这通常是因为不同页面仍在读取旧字段：

```text

ctx.client.model_name
.env 里的 MODEL_NAME
```

这些都不应该作为当前真实模型状态来源。

要求：

1. 所有当前模型显示统一从 `LLMRouter` 获取。
2. 新增或完善 `router.get_status()`，返回统一结构，例如：

```python
{
    "provider_mode": "auto",
    "manual_provider": None,
    "current_provider": "...",
    "current_model": "...",
    "last_success_provider": "...",
    "last_success_model": "...",
    "last_fallback_reason": "...",
    "providers": [...]
}
```

3. Web 顶栏 `base.html` 不再读取 `ctx.client.model_name`。
4. 仪表盘 `dashboard.py` 不再传 `model=settings.MODEL_NAME`。
5. Telegram `/status` 不再显示旧 `MODEL_NAME`，而是显示 router 当前状态。
6. 如果当前处于 fallback 状态，应显示：

   * 最近一次失败 provider；
   * fallback 到哪个 provider；
   * fallback 原因；
   * 当前实际使用模型。
7. 如果当前模型未知，不要显示旧默认值，应显示“等待首次调用”或“当前未确认”。

目标：顶栏、仪表盘、模型管理页、Telegram `/status` 对当前模型的显示必须一致。

七、修复错误分类顺序：no_channel 必须优先于 model_not_found

中转平台可能返回：

```text
model_not_found: No available channel for model xxx under group free
```

这类错误不应该被判定为“模型名错误”，而应判定为“免费通道当前不可用”。

请修改错误分类顺序：

1. 先判断：

   * no available channel
   * under group free
   * no channel
2. 再判断：

   * model_not_found
   * model does not exist
   * unknown model

处理策略：

```text
no_channel：
  冷却 10-30 分钟
  自动 fallback
  不标记余额耗尽
  不标记永久配置错误

model_not_found：
  标记配置错误或模型不存在
  Web 提示检查 model id
```

目标：VSLLM 免费模型通道不可用时，不要误判成模型名错误。
