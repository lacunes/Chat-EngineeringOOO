你现在需要对当前项目进行一次系统性迭代。

项目定位：
这是一个以 Telegram 为主要交互入口、面向单用户沉浸式角色扮演的 AI 系统。项目已经具备世界观切换、角色管理、长期记忆、关系系统、时间系统、NPC 行为、剧情状态、多模型路由和 Web 导演台等功能。

本次迭代参考 NachoBot / MaiBot 一类项目中的优秀设计思路，但禁止简单照搬或整体替换现有架构。

核心原则：
1. 保留当前项目“单用户、长篇角色扮演、剧情控制”的专门化优势。
2. 只吸收适合当前项目的机制。
3. 不引入与当前目标无关的 QQ、Discord、B站、直播、多平台适配。
4. 不引入本地大型 VLM、ASR、TTS 或重型 Agent 沙盒。
5. 不进行没有必要的大重构。
6. 优先兼容当前数据和配置文件。
7. 每一个阶段都必须保证项目可以正常启动、正常聊天和正常读取旧数据。
8. 先审查现有代码，再确定修改方案，禁止凭空假设项目结构。
9. 不得大段复制 NachoBot 源码，只能借鉴架构思路并独立实现。
10. 不得为了追求“架构先进”而破坏现有稳定功能。

==================================================
三、长期记忆系统重构
==================================================

当前长期记忆不能继续只作为无类型的摘要列表存在。

将长期记忆设计为结构化记录，但必须兼容旧 memory.json。

建议至少支持以下类型：

1. fact
稳定事实，例如身份、习惯、背景、已确认设定。

2. relationship
角色之间的关系变化，例如信任、依赖、矛盾、亲密度变化。

3. event
已经发生的重要剧情事件。

4. promise
角色之间的承诺、约定和未完成事项。

5. preference
用户或角色的稳定偏好。

6. secret
只有部分角色知道的信息。

7. goal
角色当前目标、长期目标和待完成目标。

8. scene_state
需要跨会话保留的场景状态。

建议的数据结构：

{
  "id": "mem_xxx",
  "world_id": "one",
  "type": "event",
  "content": "陈平得知林栖真实身份",
  "participants": ["陈平", "林栖"],
  "importance": 0.90,
  "confidence": 0.95,
  "status": "active",
  "tags": ["身份", "秘密"],
  "source_message_ids": [],
  "created_at": "",
  "updated_at": "",
  "last_recalled_at": "",
  "recall_count": 0,
  "expires_at": null
}

具体要求：

1. 所有记忆必须绑定 world_id。
2. 角色相关记忆必须记录 participants。
3. 记忆必须具备唯一 ID。
4. 记忆写入必须使用原子写入。
5. 禁止因为解析失败覆盖整个记忆文件。
6. 加载失败时必须保留原文件并创建错误日志。
7. 重启后记忆不能消失。
8. 切换模型后记忆不能消失。
9. 切换世界后不得串世界召回。
10. 需要提供旧格式迁移逻辑。
11. 迁移前自动创建备份。
12. 同一事实应支持合并和更新，减少重复。
13. 冲突事实不能直接互相覆盖，应记录置信度或更新时间。
14. 记忆提取失败不得影响主回复发送。
15. 记忆写入应和主聊天解耦，但不能使用不可控的后台任务。
16. 所有失败必须有清晰日志。

需要实现记忆生命周期：

- created：新建
- active：有效
- resolved：已解决，例如完成的承诺
- superseded：被新事实替代
- archived：归档
- deleted：人工删除

需要实现“誓约/承诺”管理：

- 谁对谁作出承诺
- 承诺内容
- 当前状态
- 是否已经履行
- 是否被违背
- 关联剧情事件
- 最近提及时间

==================================================
四、动态情景注入与上下文选择器
==================================================

不要继续把所有世界观、所有角色设定、所有长期记忆和全部关系状态无差别塞入提示词。

增加 Context Selector。

输入：

- 当前用户消息
- 当前 world_id
- 当前场景
- 当前在场角色
- 最近短期对话
- StoryState
- 时间状态
- 关系状态
- 长期记忆库

输出：

- 本轮相关世界观片段
- 本轮相关角色设定
- 当前场景状态
- 相关长期记忆
- 相关承诺
- 相关关系变化
- 必须保留的剧情事件
- 必要的时间信息

召回优先级建议：

1. 当前场景直接相关信息
2. 当前在场角色的信息
3. 当前消息明确提到的角色或事件
4. 活跃承诺和未完成目标
5. 最近发生的重要事件
6. 高重要度稳定事实
7. 低重要度背景信息

要求：

1. 必须设置上下文预算。
2. 不同模块拥有独立预算。
3. 不允许某个模块无限占用上下文。
4. 必须保留最近短期对话。
5. 必须避免重复注入相同信息。
6. 必须避免把同一事实同时从世界观、记忆和角色卡重复塞入。
7. 必须记录每轮选择了哪些上下文。
8. 必须记录每条记忆被召回的原因。
9. 提示词构建失败时，应回退到安全的基础上下文。
10. Context Selector 不得影响 Telegram 主流程稳定性。

建议输出调试结构：

{
  "world_context": [],
  "character_context": [],
  "memory_context": [],
  "relationship_context": [],
  "story_context": [],
  "excluded_items": [],
  "token_budget": {},
  "selection_reasons": {}
}

==================================================
五、轻量事件总线和模块解耦
==================================================

参考插件式架构的思路，但不要建立过重的插件市场或动态下载机制。

可以实现一个轻量 EventBus。

建议事件：

- before_user_message
- after_user_message
- before_context_selection
- after_context_selection
- before_prompt_build
- after_prompt_build
- before_llm_request
- after_llm_response
- after_assistant_reply
- memory_created
- memory_updated
- relationship_changed
- time_advanced
- world_changed
- provider_failed
- provider_switched
- story_state_changed

要求：

1. 主聊天流程不能依赖某个非核心监听器成功。
2. 单个监听器报错不能中断整个消息处理。
3. 每个事件必须记录错误来源。
4. 监听器必须具备明确优先级。
5. 不允许循环触发。
6. 不允许事件监听器偷偷修改不可变的核心输入。
7. 对关键上下文修改必须显式返回。
8. 保留同步执行模式，避免引入复杂后台任务。
9. 先对现有 RelationshipManager、TimeManager、NPCManager 和 MemoryManager 做最小改造。
10. 不要一次性把所有模块全部插件化。

优先插件化：

- 回复后记忆提取
- 关系更新
- 时间推进
- StoryState 更新
- NPC 主动行为
- 回复质量检查
- Provider 失败日志

==================================================
七、Web 端整体视觉整改
==================================================

当前 Web UI 存在明显的“AI 生成页面感”，需要整体整改。

目标：
将 Web 端改造成“沉浸式角色扮演系统的导演台”，而不是通用 AI 聊天工具或 AI 产品落地页。

设计参考方向：

- shadcn/ui 式的中性、清爽和可扩展结构
- Tabler 式的现代后台控制台
- Flowbite 的 minimal / enterprise 方向
- 使用统一线性 SVG 图标体系，例如 Lucide 风格
- 可以借鉴这些设计语言，但不要机械复制模板

整体气质：

- flat
- clean
- calm
- structured
- editorial
- dashboard-like
- minimal
- enterprise
- low saturation
- subtle
- readable

中文描述：

- 扁平化
- 简洁
- 克制
- 现代
- 低饱和
- 信息清晰
- 稍微有叙事系统感
- 像成熟的内部工具
- 像剧情导演台
- 不像 AI 产品首页

7.2 配色规范
--------------------------------------------------

建议使用中性灰白作为主体。

推荐基础变量：

--background: #F8FAFC;
--surface: #FFFFFF;
--surface-secondary: #F1F5F9;
--text-primary: #0F172A;
--text-secondary: #475569;
--text-muted: #64748B;
--border: #E2E8F0;
--border-strong: #CBD5E1;

只选择一个主强调色，例如：

方案一：
--accent: #3B82F6;

方案二：
--accent: #4F46E5;

方案三：
--accent: #0F766E;

要求：

1. 整个系统只保留一个主要强调色。
2. 成功、警告、错误状态色仅用于状态表达。
3. 禁止大面积渐变。
4. 禁止每个页面使用不同主色。
5. 深色模式必须保持相同的信息层级，而不是简单反色。
6. 不要用纯黑作为大面积背景。
7. 状态色应低饱和，避免荧光色。

Badge：

表格：

--------------------------------------------------
7.9 Web 技术实现原则
--------------------------------------------------

1. 先确认当前前端技术栈。
2. 不要为了换 UI 强制更换整个框架。
3. 如果现有组件体系可复用，就建立统一设计 token。
4. 如果现有 CSS 混乱，先建立：
   - color tokens
   - spacing tokens
   - radius tokens
   - typography tokens
   - shadow tokens

5. 样式应集中管理。
6. 不允许大量 inline style。
7. 不允许每个页面单独定义不同按钮和卡片。
8. 建立统一基础组件：
   - Button
   - Card
   - Input
   - Select
   - Badge
   - Dialog
   - Drawer
   - Tabs
   - Table
   - Tooltip
   - Toast
   - EmptyState
   - ErrorState
   - LoadingState

9. 旧页面逐步迁移，不要一次全部推倒重写。
10. 所有功能整改前后必须保持可用。

==================================================
八、安全、稳定性和兼容性要求
==================================================

1. 不得在前端或日志中暴露 API Key。
2. 不得通过 Web 暴露任意 shell 执行。
3. 配置保存必须先校验。
4. 配置修改必须备份。
5. JSON/YAML 写入必须原子化。
6. 数据迁移必须可回滚。
7. 修改数据库或 JSON 结构前必须创建备份。
8. 所有新字段应提供默认值。
9. 旧配置缺少字段时不得启动失败。
10. 单个附加模块报错不得影响主聊天。
11. Telegram 聊天链路优先级高于 Web 增强功能。
12. 不允许因为 Web 页面崩溃导致 Bot 主服务退出。
13. 不允许未经确认删除旧数据。
14. 不允许在重构中擅自删除现有功能。
15. 发现废弃功能时先列出，确认无调用后再删除。
16. 所有异常必须记录模块名和请求 ID。
17. 日志必须自动脱敏：
    - API Key
    - Bot Token
    - Authorization Header
    - Cookie
    - Secret
