## 4. 诊断结果

### 4.1 已确认事实

1. relations.html:60-70 内外层循环都用 `loop.index0`，内层遮蔽外层 → 维度字段名跨行错乱
2. relations.py:113 `int(request.form.get(f"rel_{i}_{dim}", "0") or 0)` 找不到字段默认 0
3. main.py:72,79,110 全部使用 `logging.FileHandler`，无轮转

### 4.2 根因

**问题1**：Jinja2 内层循环 `loop` 变量覆盖外层。6 维度 × N 行场景下，每行只有 1 个维度字段名碰巧正确（当外层 index == 内层 index 时），其余 5 个都映射到错误行号，后端找不到即归零。

**问题2**：FileHandler 不限制大小，日志文件无限增长。

## 11. 修改结果说明

### 11.1 修改摘要

1. **relations.html**：外层循环中保存行索引 `row_index`，维度字段 `rel_{{ row_index }}_{{ dim }}` 不受内层循环干扰
2. **main.py**：所有 FileHandler 替换为 RotatingFileHandler，app.log 5MB×5，error.log 2MB×10，专项日志 2MB×5
3. **clean_logs.sh**：安全清理 `*.log.*` 轮转备份，保留当前 `.log` 文件
4. **test_relations_save.py**：4 个测试覆盖保存/更新/新增/删除场景

### 11.2 关键逻辑变化

#### 文件：web/templates/relations.html

- 原逻辑：所有字段直接使用 `loop.index0`，内外层冲突
- 新逻辑：外层循环开头 `{% set row_index = loop.index0 %}`，所有字段引用 `row_index`
- 修改原因：Jinja2 内层 `loop` 遮蔽外层，维度字段名映射到错误行号
- 潜在影响：无

#### 文件：main.py

- 原逻辑：`logging.FileHandler` 直接写文件，无大小限制
- 新逻辑：`logging.handlers.RotatingFileHandler` 按 maxBytes 自动轮转
- 修改原因：防止日志无限增长
- 潜在影响：日志文件变大时自动切分，旧日志在 `.log.1`/`.log.2` 等备份中
