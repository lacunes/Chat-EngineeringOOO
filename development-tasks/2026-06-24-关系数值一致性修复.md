## 2. 需求理解

### 2.1 AI 对目标的理解

- 要解决的问题：关系“当前总值”和“变化量”在 Web、Prompt、Telegram、内存及 JSON 之间语义混淆或时序不一致。

- 本次范围：数据链路审计、根因定位、关系提示展示渠道调整、关系日志、revision/异步冲突策略、Web 只读调试信息、关系数字输出警告、短期记忆污染只读检查脚本、自动测试与文档同步。

## 4. 诊断结果

### 4.1 已确认事实

1. `AppContext.relationship_manager` 直接返回 `RoleplayBot.relationship_manager`，Web、Telegram、ContextSelector 使用同一实例。
2. 主模型原始正文先写入短期记忆；随后 `take_pending_hints()` 才把上一轮后台抽取提示拼到本轮 Telegram 最终文本开头。
3. `auto_extract()` 以 `len(memory)` 作为 `message_index`，未记录启动 revision、完成时间或稳定消息 ID。
4. Web 和 AI 修改受同一 `RLock` 保护，但锁只防止同时写入，不能识别等待 API 期间发生的 Web 新修改。
5. 当前 `relation.log` 只记录非结构化摘要/差异；Web 手动修改使用普通路由 logger，不记录逐维度 before/after。
6. `RELATION_INJECT_INSTRUCTION` 仅有一句弱约束，模型仍可能输出关系面板式数字。
7. 修改前相关基线测试 `23 passed`。

### 4.2 高可能原因

1. 用户在 Telegram 开头看到的变化数字来自 `_pending_hints`，且由于后台任务异步完成，只会在下一条回复被取出，造成与当前轮叙事错位。
2. 正文中的“当前好感 75”等数字来自主模型对 Prompt 内部数值的复述，弱提示无法完全抑制。
3. Web 显示值与 Prompt/JSON 若出现不一致，当前没有写后自动校验或调试元数据可快速证明具体分层。

### 4.3 待验证推测

1. 旧短期记忆中可能已有模型自行输出的关系面板式数字；只能通过只读扫描确认，不自动清理。

### 4.4 已排除项

| 已排除可能 | 排除依据 |
|---|---|
| Web 和 Telegram 默认使用两个独立 RelationshipManager 实例 | `AppContext` 属性直接代理 `RoleplayBot` 实例 |
| 程序拼接的 pending hint 被当前代码写入短期记忆 | `add_assistant_message(reply)` 在拼接逻辑之前执行 |
| 旧异步结果直接按旧基线覆盖成 62 | `apply_changes` 对应用时当前内存值做加法；实际风险是过时 delta 被继续应用成 82 |

## 5. 方案比较

### 5.1 方案 A（选定）

- 目标：过时抽取结果安全丢弃，关系变更全链路可审计。
- 修改深度：为 RelationshipManager 增加持久化 revision、变更元数据、一致性校验和结构化变更记录；移除 Telegram pending hint 展示；增加只读 Web 调试卡和扫描脚本。
- 优点：行为安全、可解释，不会把旧对话的 delta 叠加到 Web 新设定；不改变现有关系 JSON 主体结构。
- 缺点：发生并发 Web 修改时，本次自动抽取结果会丢失。
- 风险：旧测试桩需适配新接口；新增元数据必须保持旧 JSON 兼容。
- 是否推荐：是。

### 5.2 方案 B（不选）

- 目标：revision 变化后仍把旧 delta 重新叠加到当前值。
- 优点：不丢失自动抽取结果。
- 缺点：无法证明旧对话 delta 在用户手动改值后仍有效，解释性和安全性较差。
- 是否推荐：否。

## 11. 修改结果说明

### 11.1 修改摘要

1. 取消关系变化提示混入 Telegram 正文，最终发送文本等于主模型原始正文。
2. Web/AI 有效修改统一递增 revision；旧异步抽取版本不匹配时丢弃。
3. `relation.log` 逐维度记录 `before`、带符号 `delta`、`after` 与来源。
4. 写入后立即回读 JSON 校验；Web 展示共享实例及下一轮 Prompt 摘要。
5. 新增只读短期记忆关系数字扫描脚本。

### 11.2 关键逻辑变化

- 原逻辑：后台抽取把 `好感+2` 放入 pending，下轮 Telegram 回复开头取出；无版本基线。
- 新逻辑：抽取只更新内部状态并记录日志；启动 revision 与 apply revision 不一致则丢弃。
- 原逻辑：Web 路由直接改字典后 `save()`，没有逐维度审计元数据。
- 新逻辑：Web 在共享 manager 锁内建立快照并统一 commit，写 revision、来源、时间和变化记录。

### 11.3 关键 Diff

```diff
- pending = relationship_manager.take_pending_hints()
- reply = "（" + "；".join(pending) + "）\n\n" + reply
+ relationship_manager.warn_if_reply_exposes_relation_numbers(reply)
+ # 原始 reply 写入短期记忆并原样发送

+ if expected_revision != self.revision:
+     logger.warning("stale relation extraction discarded ...")
+     return None

+ [RELATION_CHANGE] ... before=60 delta=+2 after=62
```

### 11.4 值得学习的内容

1. 锁只能阻止同时写入，不能判断异步调用等待期间的业务状态是否已经过期；revision 解决的是后者。
2. 当前值与 delta 必须在类型、变量名和日志中分开，不能靠文案猜语义。
3. Prompt 约束与输出警告各有职责：前者预防，后者观测；不应粗暴删除普通叙事数字。
