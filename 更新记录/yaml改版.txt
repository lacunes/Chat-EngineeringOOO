请把当前 data/worlds/*.json 世界数据格式迁移为 YAML 格式，以提升手动编辑体验。

背景：
当前世界数据已经从 bot/worlds/*.py 迁移到 data/worlds/*.json。
JSON 稳定，但长文本字段会变成一整行，通过 \n 表示换行，手动编辑体验很差。
我希望保留“结构化数据文件”的方向，但改成更适合长文本编辑的 YAML。

目标：
1. 使用 data/worlds/*.yaml 作为世界数据主来源。
2. 不要恢复 Web 直接编辑 bot/worlds/*.py。
3. 不要修改世界文本内容。
4. 不要改变聊天逻辑。
5. 不要改变 prompt 结构。
6. 不要增加 API 调用。
7. 保留旧 json 的兼容读取能力，至少过渡期不要直接删除。
8. Web 世界编辑器应读取和保存 YAML。
9. 长文本字段在 YAML 中使用块文本格式，方便手动编辑。
10.已经废弃的东西可以删除掉，如原“one.py”等及其文件夹

建议 YAML 结构：

WORLD_NAME: one

START_SCENE: |
  多行开场文本

SYSTEM_PROMPT: |
  多行系统提示词

CHARACTERS:
  角色名: >
    角色描述

RULES: []

LOCATIONS: {}

EVENT_POOL: []

NPCS: {}

请重点修改：
- world_loader.py 或当前世界加载逻辑
- web/routes/worlds.py 或当前 Web 世界编辑器逻辑
- docs/OPERATIONS.md
- docs/ARCHITECTURE.md
- docs/DEVELOPMENT_RULES.md

具体要求：

一、YAML 加载

1. 新增对 data/worlds/*.yaml 的读取。
2. 优先读取 yaml/yml。
3. 如果 yaml 不存在，再读取旧 json。
4. 如果 json 存在但 yaml 不存在，可以提供迁移函数，但不要自动破坏原 json。
5. 字段兼容：
   - WORLD_NAME / world_name
   - START_SCENE / start_scene
   - SYSTEM_PROMPT / system_prompt
   - CHARACTERS / characters
   - RULES / rules
   - LOCATIONS / locations
   - EVENT_POOL / event_pool
   - NPCS / npcs

二、YAML 保存

1. Web 面板保存时写入 data/worlds/<name>.yaml。
2. 保存前校验必要字段。
3. 保存时使用原子写入：
   - 先写临时文件
   - 校验能重新读取
   - 再替换正式文件
4. 保存失败不能破坏原文件。
5. 保存前可以备份旧 yaml 到 backups/worlds/，但如果实现复杂，可以先不做。

三、长文本格式

保存 YAML 时：
1. START_SCENE 使用块文本风格，类似：
   START_SCENE: |
     第一行
     第二行

2. SYSTEM_PROMPT 使用块文本风格：
   SYSTEM_PROMPT: |
     第一行
     第二行

3. 角色描述可以用普通字符串或折叠文本，但要尽量保持可读性。
4. 不要把长文本重新保存成带大量 \n 的一整行。

四、依赖

1. 如果项目已经有 PyYAML，直接使用。
2. 如果没有，请在 requirements.txt 增加 PyYAML。
3. 不要引入复杂配置库。

五、文档更新

docs/OPERATIONS.md：
- 说明以后手动编辑世界应编辑 data/worlds/*.yaml。
- 说明 json 是旧兼容格式。
- 说明不要再编辑 bot/worlds/*.py 作为主世界入口。

docs/ARCHITECTURE.md：

docs/DEVELOPMENT_RULES.md：

- 修改世界数据时优先处理 data/worlds/*.yaml。

特别注意 Web 端体验：

1. 改成 YAML 后，Web 面板的表单体验不能倒退。
2. Web 端仍然应该以输入框/大文本框/列表的方式编辑世界，而不是直接让用户编辑原始 YAML。
3. START_SCENE 和 SYSTEM_PROMPT 在 Web 表单中仍然是多行 textarea。
4. 保存到 YAML 时，START_SCENE 和 SYSTEM_PROMPT 必须使用 | 块文本风格，不能保存成带 \n 的一整行。
5. 不要使用复杂 YAML 特性，例如锚点、引用、tag。
6. 保存失败不能破坏原 YAML 文件。
7. 如果 YAML 解析失败，Web 页面要显示清晰错误，不要白屏。
