二、/c 续写不要触发后台维护

请检查 /c 或 continue_story 的实现。

我的需求是：

/c 只是让 AI 延续当前回复，不应该被当成一次完整的新对话轮次处理。

因此 /c 不应该触发：

1. 长期记忆抽取
2. 长期记忆精炼
3. 关系网络抽取
4. 关系抽取计数递增
5. 时间推进
6. 后台维护任务

请设计：

1. 给 generate_reply 增加参数：
   skip_background_maintenance: bool = False

2. 普通聊天：
   skip_background_maintenance=False

3. /c 续写：
   skip_background_maintenance=True

4. skip_background_maintenance=True 时：
   - 仍然保存必要的短期上下文
   - 仍然正常回复用户
   - 不触发维护类 API 调用
   - 不影响后续正常聊天

三、关系网络抽取前增加本地必要性判断

关系网络抽取是后台维护功能，不需要每次都调用 API。

请在调用关系抽取 API 前，增加一个保守的本地判断函数，例如：

should_extract_relationships(recent_messages, characters)

判断条件可以包括：

1. 最近对话里是否出现两个及以上角色名
2. 是否出现关系变化相关关键词，例如：
   信任、怀疑、喜欢、讨厌、害怕、依赖、承诺、背叛、保护、生气、道歉、亲近、疏远、试探、安慰、嫉妒、吃醋、照顾、冷淡、和好
3. 是否出现明显情绪冲突或亲密互动
4. 是否出现新的承诺、关系确认、关系破裂、关系缓和

如果没有明显信号，就跳过关系抽取 API。

要求：

1. 判断要保守。
2. 宁可漏掉轻微变化，也不要频繁调用 API。
3. 本地判断只决定是否调用 API，不直接修改关系数值。
4. 增加配置项：
   RELATION_EXTRACT_REQUIRE_SIGNAL = True

四、长期记忆抽取前增加本地必要性判断

长期记忆也不需要每轮都抽取。

请在调用长期记忆抽取 API 前，增加一个本地判断函数，例如：

只有出现较明确的长期信息时才调用 API，例如：

1. 新的人物关系
2. 新承诺
3. 新习惯
4. 明确偏好
5. 重要事件
6. 角色阶段变化
7. 生活状态变化
8. 用户明确设定

   - 本地判断存在抽取价值

3. 本地判断失败时只跳过 API，不影响聊天。

六、压缩旧记忆后不要强制触发长期记忆精炼

请检查 MemoryManager.compress_old_memory()。

如果压缩旧短期记忆已经调用一次 API 生成摘要，之后又强制调用 refine_long_memory，可能导致一次维护产生两次 API 调用。

请优化为：

1. 压缩旧短期记忆后，只添加“旧剧情摘要”到长期记忆。
2. 不要默认 force refine。
3. 只有长期记忆数量明显超过上限时，才触发精炼。
4. 增加配置：
   LONG_MEMORY_REFINE_BUFFER = 20
5. 只有当长期记忆数量超过 LONG_MEMORY_MAX_ITEMS + LONG_MEMORY_REFINE_BUFFER 时才精炼。
