请阅读当前项目，重点分析并小幅优化人物关系系统。

项目背景：
这是一个 Telegram AI Roleplay Framework，已有长期记忆、短期上下文、NPC 主动行为、动态状态、世界观切换等功能。现在希望让角色关系更加稳定、可追踪，但不能显著增加 API 开销。

本次目标：
建立“结构化关系状态”的基础能力，让角色关系变化更加清晰，但不要每轮都额外调用关系分析 API。

非常重要：
1. 不要增加每轮主聊天的 API 调用次数。
2. 不要每轮都调用关系分析。
3. 不要减少短期上下文。
4. 不要删除或弱化长期记忆。
5. 不要关闭 NPC 主动行为。
6. 不要修改角色人设文本。
7. 不要大规模重构。
8. 优先兼容现有数据结构。

请重点查看：
- bot/relations.py
- bot/memory_manager.py
- bot/telegram_handlers.py
- bot/worlds/*.py
- bot/settings.py
- Web 面板里和关系网络相关的文件
- 任何 relation_extract、relationship、dynamic_state 相关代码

请先分析：
1. 当前是否已有关系系统？
2. 关系信息目前从哪里来？
3. 关系摘要是否会进入主聊天 prompt？
4. 关系更新是否依赖 API？
5. 是否每轮都会调用关系分析？
6. 当前关系状态是否有结构化字段？

然后进行低风险优化：

一、设计轻量结构化关系模型

建议关系字段如下：

- from_character
- to_character
- intimacy：亲密度，0-100
- trust：信任度，0-100
- dependency：依赖度，0-100
- tension：冲突/紧张度，0-100
- possessiveness：占有欲，0-100，可选
- label：当前关系标签
- summary：一句话关系摘要
- last_event：最近导致关系变化的事件
- updated_at：更新时间

如果现有系统不适合完整改造，可以先用兼容方式实现，例如 dict/json 文件，不要强行改数据库。

二、关系分析不要每轮触发

请采用低频或条件触发策略，例如：
- 每隔 N 轮才分析一次
- 或者只有最近对话出现明显关系变化关键词时才触发
- 或者由用户/后台手动触发
- 或者沿用项目已有触发机制，不提高频率

请不要实现：
主聊天每轮结束后必定调用 relation_extract。

三、关系状态进入 dynamic_state

请确保主聊天 prompt 中可以看到简洁关系摘要，例如：

[当前关系状态]
林栖 -> 陈平：亲密度 78，信任 91，依赖 86，关系：强依赖/亲密。最近变化：林栖在危险后主动寻求陈平安慰。

要求：
1. 关系摘要要简短。
2. 不要把整张关系表都塞进 prompt。
3. 只塞当前世界、当前相关角色的重要关系。
4. 不要替代长期记忆，只作为动态状态补充。

四、关系更新要谨慎

关系分析 prompt 应要求模型：
1. 只在发生明确互动后调整数值。
2. 一次变化幅度不要过大，除非发生重大剧情事件。
3. 不要根据模型猜测更新关系。
4. 不要把玩笑话当作重大关系变化。
5. 保留 last_event 作为依据。
6. 如果没有明显变化，返回 no_change。

五、日志

增加关系分析日志：
- 是否触发关系分析
- 触发原因
- 是否 no_change
- 哪些关系字段变化
- 变化前后数值
- 本次是否调用 API

六、成本控制

请明确检查：
1. 本次修改是否增加每轮 API 调用？

3. 是否可以用便宜模型处理关系分析？
4. 是否可以关闭或调低频率？
