
项目背景：
这是一个挂载在 Telegram 上的 AI 角色扮演聊天机器人。它通过 Telegram 接收用户消息，调用 DeepSeek API 生成回复，并支持世界观、角色、记忆、关系、时间、NPC 主动行为等功能。

我希望你重点升级 Web 管理端。

Web 端的定位：
它不是 VPS 控制台，也不是系统运维后台，而是“AI 角色扮演导演台”。

我希望以后可以通过外网 Web 页面完成大部分剧情和机器人行为调整，例如：
- 修改世界设定
- 修改角色设定
- 修改长期记忆
- 检查记忆污染
- 修改人物关系
- 控制剧情时间
- 调整回复长度、上下文、NPC 主动行为等参数
- 查看项目日志和运行状态

但是系统级操作仍然保留在 VPS/SSH 中完成。

非常重要的边界：
1. 不要做网页终端。
2. 不要做维护中心。
3. 不要提供 git pull、pip install、systemctl restart、重启 Bot 等功能。
4. 不要执行任何用户输入的 shell 命令。
5. 不要展示 BOT_TOKEN、DEEPSEEK_KEY、WEB_PASSWORD、ALLOWED_ID、WEB_HOST、WEB_PORT。
6. 敏感配置不要显示，连“已设置 / 未设置”也不要显示。
7. 不要破坏现有 Telegram Bot 核心聊天逻辑。
8. 不要破坏现有 worlds、memory、relationship、time 的数据格式。
9. Web 端需要支持外网访问，因此必须强化登录保护和基础安全措施。
10. 不要使用外部 CDN，CSS 和 JS 全部本地化。

2. 配置中心

配置中心只管理“剧情和机器人行为参数”，不管理密钥、网络、安全、系统参数。

页面上必须用中文名称和中文说明，不要直接把 .env 变量名显示出来。

例如，不要显示：
CONTEXT_LENGTH
SPLIT_THRESHOLD
NPC_BASE_ACTIVATION

而要显示：
- 最近对话保留数量
- 自动分段阈值
- NPC 主动出现概率

每个配置项都要有：
- 中文名称
- 当前值
- 推荐范围
- 简短说明
- 输入校验

可管理项目包括：

1）当前世界
内部对应 ACTIVE_WORLD
说明：选择当前启用的世界设定。
要求：只能选择 worlds/ 目录下存在的世界。

2）使用模型
内部对应 MODEL_NAME
说明：选择 DeepSeek 模型。
建议选项：deepseek-chat、deepseek-reasoner。

3）回复最短长度
内部对应 MIN_REPLY_TOKENS
说明：控制 AI 单次回复的最低长度，越高越不容易短回复。
推荐范围：300-700。

4）回复常规长度
内部对应 MID_REPLY_TOKENS
说明：大多数情况下 AI 会接近这个长度。
推荐范围：1000-2400。

5）回复最长长度
内部对应 MAX_REPLY_TOKENS
说明：限制单次生成的最大长度。
推荐范围：1800-3600。

6）自动分段阈值
内部对应 SPLIT_THRESHOLD
说明：超过该长度时自动拆成多条 Telegram 消息。
推荐范围：1500-3000。

7）续写最大次数
内部对应 CONTINUE_LIMIT
说明：使用 /continue 时最多连续续写几次。
推荐范围：1-5。

8）最近对话保留数量
内部对应 CONTEXT_LENGTH
说明：每次请求模型时携带最近多少条聊天记录。太低会失忆，太高会浪费 token，也可能让模型抓不住重点。
推荐范围：40-80。

9）短期记忆最大容量
内部对应 MEMORY_MAX_LENGTH
说明：本地短期记忆最多保存多少条。
推荐范围：100-300。

10）长期记忆参考上限
内部对应 LONG_MEMORY_CONTEXT_LIMIT
说明：每次回复时最多引用多少条长期记忆。
推荐范围：10-40。

11）长期记忆总量上限
内部对应 LONG_MEMORY_MAX_ITEMS
说明：长期记忆最多保留多少条。
推荐范围：100-500。

12）自动整理记忆间隔
内部对应 AUTO_MEMORY_INTERVAL
说明：每隔多少轮对话尝试提取一次长期记忆。太小可能造成记忆污染，太大可能漏掉重要剧情。
推荐范围：8-15。

13）记忆回看范围
内部对应 AUTO_MEMORY_LOOKBACK
说明：自动整理记忆时回看最近多少条对话。
推荐范围：20-60。

14）手动记忆长度限制
内部对应 MEMO_SIZE_LIMIT
说明：/memo 手动添加记忆时允许的最大长度。
推荐范围：200-1000。

15）NPC 主动出现概率
内部对应 NPC_BASE_ACTIVATION
说明：NPC 在合适场景中主动介入的基础概率。越高，世界越热闹；越低，剧情越围绕当前对话双方。
推荐范围：0.15-0.35。

16）单次最多主动 NPC 数
内部对应 NPC_MAX_ACTIONS_PER_CHECK
说明：一轮中最多允许几个 NPC 主动行动。
推荐范围：1-3。

17）NPC 检查间隔
内部对应 NPC_TIMER_INTERVAL
说明：每隔多久检查一次 NPC 是否主动行动。
推荐范围：30-180。

18）关系自动分析间隔
内部对应 RELATION_EXTRACT_INTERVAL
说明：每隔多少轮对话尝试分析一次人物关系变化。太小容易被一两句话误导。
推荐范围：8-15。

19）是否允许自动推进时间
内部对应 TIME_AUTO_ADVANCE_ENABLED
说明：AI 是否能根据剧情自然推进时段。

20）是否允许用户话语推动时间
内部对应 TIME_USER_DRIVEN_ADVANCE_ENABLED
说明：当用户说“第二天”“晚上”等词时，是否自动更新时间。

21）长场景提醒阈值
内部对应 TIME_LONG_SCENE_HINT_THRESHOLD
说明：同一时段持续太久时提醒是否推进时间。
推荐范围：8-20。

配置中心要求：
- 保存 .env 前自动备份为 .env.backup.YYYYMMDD-HHMMSS。
- 保留 .env 中未知字段和注释。
- 只替换被管理字段。
- 保存后提示“配置已保存，部分配置可能需要重新启动进程后生效”。
- 不要提供重启按钮。

5. 长期记忆污染检查

请新增“长期记忆污染检查”功能。

目标：
帮助我检查长期记忆中是否存在重复、矛盾、过时、幻觉风险、过细无价值、表述不清等问题。

检查类型：
- duplicate：重复
- contradiction：矛盾
- outdated：过时
- hallucination_risk：幻觉风险
- too_trivial：过细无价值
- unclear_subject：主语不清
- too_long：过长
- too_short：过短

重要要求：
1. 不要自动修改长期记忆。
2. AI 只能生成检查报告和修改建议。
3. 所有删除、合并、改写都必须由用户在 Web 页面二次确认。
4. 支持规则检查 + AI 检查两层。
5. 规则检查负责：
   - 完全重复
   - 过短
   - 过长
   - 含糊词
   - 高相似文本
6. AI 检查负责：
   - 语义重复
   - 矛盾
   - 过时
   - 幻觉风险
   - 表述不清
7. AI 检查要分批处理长期记忆，每批 20-40 条。
8. AI 输出必须要求为 JSON，并处理解析失败。
9. 每次检查生成报告，保存在 memory_audit/ 目录。
10. 报告文件名格式：
    memory-audit-YYYYMMDD-HHMMSS.json
11. 页面需要显示检查历史，可以重新打开旧报告。
12. 报告中要显示：
    - 涉及记忆编号
    - 原文
    - 问题类型
    - 问题说明
    - 建议操作
    - 建议合并文本或改写文本
13. 页面操作包括：
    - 采用建议
    - 手动编辑
    - 删除某条
    - 忽略此问题

6. 关系网络

请优化关系网络系统。

当前关系系统支持 0-100 和 110 锁死上限。现在我希望加入 -100 作为锁死下限。

规则：
- 普通可变范围：0-100
- 110：该维度锁死极高，自动关系分析不得降低
- -100：该维度锁死极低，自动关系分析不得提高
- Web 表单只允许输入 -100、0-100、110
- 自动关系更新时，110 和 -100 都必须保持锁死
- 其他值只能在 0-100 范围内变化
- 不同维度含义不同，不要把 110 统一解释成“好”
- 110 表示该维度极高，例如好感极高、敌意极高、怀疑极高
- -100 表示该维度极低，例如好感锁死极低、敌意锁死极低、怀疑锁死极低

新建关系默认值：
affection: 60
trust: 50
dependence: 30
fear: 0
suspicion: 20
hostility: 0

Web 页面要用中文解释各维度：
- 好感
- 信任
- 依赖
- 恐惧
- 怀疑
- 敌意

页面要解释：
- 110 = 锁死极高
- -100 = 锁死极低
- 0-100 = 普通范围

9. 外网访问安全

因为我需要外网访问 Web 页面，请加强基础安全。

要求：
- 登录保护
- session 登录
- 登录失败次数限制
- 连续失败 3 次后冷却 30 分钟

- 所有 POST 表单必须带 CSRF token
- session cookie 设置 httponly、samesite=lax
- 支持 HTTPS 反代后的 secure cookie 配置
- 安全响应头：
  - X-Frame-Options: DENY
  - X-Content-Type-Options: nosniff
  - Referrer-Policy: no-referrer
  - Content-Security-Policy: default-src 'self'
- 未登录访问任何后台页面都跳转到登录页
- 登录页不要显示项目内部路径
- 不要展示敏感信息

操作审计日志：
请记录以下操作：
- 登录成功
- 登录失败
- 保存配置
- 编辑世界
- 删除世界
- 编辑记忆
- 删除记忆
- 精炼记忆
- 运行记忆污染检查
- 采用记忆污染修复建议

- 编辑时间

审计日志可以写入 web_audit.log。

三、开发方式要求

请你先完整阅读项目结构，再制定修改方案，然后直接修改代码。

修改时请注意：
1. 不要破坏现有 Bot 聊天功能。
2. 不要破坏现有数据格式。
3. 不要删除现有功能，除非它属于系统运维类功能。
4. 尽量小步修改，保证每一步都能运行。
5. 如果现有代码结构和我描述的不完全一致，请优先兼容现有结构，而不是强行重写所有东西。

7. 不要引入复杂前端框架，优先使用 Flask + Jinja2 + 原生 CSS/JS。
8. 不要使用外部 CDN。
9. 不要把 API key、Bot token、密码写进代码或页面。
