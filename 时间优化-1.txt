请重新审视当前时间流逝系统的推进逻辑。

我的使用方式不是快节奏剧情推进，而是长时间和角色正常聊天，观察角色的日常关系变化和人生走向。

因此，一个时间段内停留几十轮甚至接近百轮是正常的。例如“深夜谈心”“客厅闲聊”“饭桌聊天”都可能持续很久。

所以我不希望时间系统机械地按照 AI 回复次数自动推进。

请把时间系统优化成“用户驱动 + 轻自动提示”的模式。

核心原则：

1. 默认不要机械自动推进时间。
   不要因为 AI 回复了 N 次，就必然推进到下一个时段。

2. /next_time 和 /next_day 仍然保留，作为明确手动推进。

3. 用户自然语言中出现明显时间跳转时，可以推进时间。
   例如：
   - 第二天早上
   - 到了晚上
   - 吃完饭后
   - 过了一会儿
   - 几天后
   - 她睡醒后
   - 陈平下班回来
   - 夜深了
   - 天亮后

4. 用户消息没有明确时间跳转时，不推进时间。
   例如：
   - 普通对话
   - 追问
   - 角色台词
   - /c 续写
   - 用户只是让角色继续说话

5. 如果同一个时间段停留很久，不要自动推进。
   可以只给一个温和提示，例如：
   “当前时段已持续较久，如需推进可使用 /next_time。”
   但不要自动改变 day 或 time_period。

6. 增加配置项：
   TIME_AUTO_ADVANCE_ENABLED = False
   TIME_USER_DRIVEN_ADVANCE_ENABLED = True
   TIME_LONG_SCENE_HINT_THRESHOLD = 80
   TIME_AUTO_CROSS_DAY = False

7. 保留 assistant_reply_count 或 message_count 只用于统计当前时段持续了多久，不再作为强制推进依据。

8. 增加一个轻量函数：
   detect_time_advance_from_user_message(user_text)

   它可以先用规则关键词判断，不要一开始就调用 AI。
   返回结果类似：
   {
     "should_advance": true,
     "target": "next_day_morning",
     "reason": "用户提到第二天早上"
   }

9. 这个检测要保守。
   宁可不推进，也不要误推进。
   只有用户文本明确出现时间变化时才推进。

10. 如果检测到“第二天早上 / 天亮后 / 次日清晨”，才允许跨天。
    普通自动推进不允许从深夜跨到第二天。

11. /c 续写永远不推进时间。

12. prompt 中注入的时间摘要仍然保留。
    时间系统只是提供生活背景，不要强推剧情。
